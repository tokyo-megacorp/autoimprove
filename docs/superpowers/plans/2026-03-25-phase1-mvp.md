# autoimprove Phase 1 MVP Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the MVP Grind loop — an autonomous experiment loop that modifies code, evaluates against deterministic benchmarks, and keeps or discards changes via git worktree isolation.

**Architecture:** Claude Code plugin with markdown skills/agents/commands. A single `evaluate.sh` script (the prepare.py equivalent) handles all deterministic evaluation. Claude orchestrates the loop, spawns experimenter agents into worktrees, and manages state in flat files.

**Tech Stack:** Claude Code plugin (markdown + YAML frontmatter), Bash 4+, jq, git worktrees

**Spec:** `docs/superpowers/specs/2026-03-25-phase1-mvp-design.md`

`scripts/evaluate.sh`:
```bash
#!/usr/bin/env bash
# scripts/evaluate.sh — autoimprove's prepare.py
# Runs gates, benchmarks, extracts metrics, applies set-logic scoring.
# Requires: jq, bash 4+
#
# Usage: evaluate.sh <config.json> [baseline.json]
#   config.json  — evaluate-config generated by orchestrator
#   baseline.json — rolling baseline (omit or /dev/null for init mode)
set -euo pipefail

CONFIG="$1"
BASELINE="${2:-/dev/null}"
INIT_MODE=false

if [ "$BASELINE" = "/dev/null" ] || [ ! -s "$BASELINE" ]; then
  INIT_MODE=true
fi

# --- Gate Runner ---
run_gates() {
  local gates_json="[]"
  local gate_count
  gate_count=$(jq '.gates | length' "$CONFIG")

  for ((i = 0; i < gate_count; i++)); do
    local name command exit_code duration_ms
    name=$(jq -r ".gates[$i].name" "$CONFIG")
    command=$(jq -r ".gates[$i].command" "$CONFIG")

    local start_ms
    start_ms=$(date +%s%3N 2>/dev/null || python3 -c 'import time; print(int(time.time()*1000))')

    if eval "$command" >/dev/null 2>&1; then
      exit_code=0
    else
      exit_code=$?
    fi

    local end_ms
    end_ms=$(date +%s%3N 2>/dev/null || python3 -c 'import time; print(int(time.time()*1000))')
    duration_ms=$((end_ms - start_ms))

    local passed="true"
    if [ "$exit_code" -ne 0 ]; then
      passed="false"
    fi

    local gate_entry
    gate_entry=$(jq -n \
      --arg name "$name" \
      --argjson passed "$passed" \
      --argjson exit_code "$exit_code" \
      --argjson duration_ms "$duration_ms" \
      '{name: $name, passed: $passed, exit_code: $exit_code, duration_ms: $duration_ms}')

    gates_json=$(echo "$gates_json" | jq --argjson entry "$gate_entry" '. + [$entry]')

    # Fast-fail on first gate failure
    if [ "$passed" = "false" ]; then
      local failed_name
      failed_name=$(echo "$gate_entry" | jq -r '.name')
      jq -n \
        --arg verdict "gate_fail" \
        --arg reason "gate '$failed_name' failed with exit code $exit_code" \
        --argjson gates "$gates_json" \
        '{verdict: $verdict, reason: $reason, gates: $gates, metrics: {}, improved: [], regressed: [], verdict_logic: "gate_fast_fail"}'
      return 1
    fi
  done

  # Return gates JSON for use by caller
  echo "$gates_json"
  return 0
}

# --- Main ---
gates_result=$(run_gates) || {
  # Gate failure — run_gates already printed the verdict JSON
  echo "$gates_result"
  exit 0
}

# If we get here, all gates passed. gates_result is the gates array.
# For now, output a minimal result (benchmarks added in next task)
if [ "$INIT_MODE" = "true" ]; then
  jq -n \
    --argjson gates "$gates_result" \
    '{mode: "init", gates: $gates, metrics: {}}'
else
  jq -n \
    --argjson gates "$gates_result" \
    '{verdict: "neutral", reason: "no benchmarks configured", gates: $gates, metrics: {}, improved: [], regressed: [], verdict_logic: "no_benchmarks"}'
fi
```

# --- Metric Extraction ---
extract_metric() {
  local output="$1" extract_pattern="$2"

  if [[ "$extract_pattern" == json:* ]]; then
    local jq_path="${extract_pattern#json:}"
    echo "$output" | jq -r "$jq_path"
  else
    # Treat as shell command (grep, awk, etc.) applied to the output
    echo "$output" | eval "$extract_pattern"
  fi
}

# --- Benchmark Runner ---
run_benchmarks() {
  local metrics_json="{}"
  local bench_count
  bench_count=$(jq '.benchmarks | length' "$CONFIG")

  for ((i = 0; i < bench_count; i++)); do
    local command
    command=$(jq -r ".benchmarks[$i].command" "$CONFIG")

    # Run benchmark, capture output
    local output
    output=$(eval "$command" 2>&1) || true

    # Extract each metric
    local metric_count
    metric_count=$(jq ".benchmarks[$i].metrics | length" "$CONFIG")

    for ((j = 0; j < metric_count; j++)); do
      local metric_name extract_pattern
      metric_name=$(jq -r ".benchmarks[$i].metrics[$j].name" "$CONFIG")
      extract_pattern=$(jq -r ".benchmarks[$i].metrics[$j].extract" "$CONFIG")

      local value
      value=$(extract_metric "$output" "$extract_pattern")

      # Store as number if possible
      if [[ "$value" =~ ^-?[0-9]+\.?[0-9]*$ ]]; then
        metrics_json=$(echo "$metrics_json" | jq --arg k "$metric_name" --argjson v "$value" '. + {($k): $v}')
      else
        metrics_json=$(echo "$metrics_json" | jq --arg k "$metric_name" --arg v "$value" '. + {($k): $v}')
      fi
    done
  done

  echo "$metrics_json"
}
```

      local name direction tolerance significance
      name=$(jq -r ".benchmarks[$i].metrics[$j].name" "$CONFIG")
      direction=$(jq -r ".benchmarks[$i].metrics[$j].direction" "$CONFIG")

      # Per-metric overrides, fallback to global
      tolerance=$(jq -r ".benchmarks[$i].metrics[$j].tolerance // .regression_tolerance" "$CONFIG")
      significance=$(jq -r ".benchmarks[$i].metrics[$j].significance // .significance_threshold" "$CONFIG")

      local baseline_val candidate_val
      baseline_val=$(echo "$baseline_metrics" | jq -r ".$name // empty")
      candidate_val=$(echo "$candidate_metrics" | jq -r ".$name // empty")

      # Skip if either value is missing
      if [ -z "$baseline_val" ] || [ -z "$candidate_val" ]; then
        continue
      fi

      # Calculate delta percentage
      local delta_pct
      if [ "$baseline_val" = "0" ]; then
        if [ "$candidate_val" = "0" ]; then
          delta_pct="0"
        else
          delta_pct="1"  # 100% change from zero
        fi
      else
        delta_pct=$(echo "scale=6; ($candidate_val - $baseline_val) / $baseline_val" | bc -l 2>/dev/null || echo "0")
      fi

      # Normalize: positive delta = improvement regardless of direction
      local normalized_delta="$delta_pct"
      if [ "$direction" = "lower_is_better" ]; then
        normalized_delta=$(echo "scale=6; -1 * $delta_pct" | bc -l 2>/dev/null || echo "0")
      fi

      # Store detail
      details=$(echo "$details" | jq \
        --arg name "$name" \
        --argjson baseline "$baseline_val" \
        --argjson candidate "$candidate_val" \
        --arg delta_pct "$delta_pct" \
        --arg direction "$direction" \
        '. + {($name): {baseline: $baseline, candidate: $candidate, delta_pct: ($delta_pct | tonumber), direction: $direction}}')

      # Check regression (normalized_delta < -tolerance)
      local is_regression
      is_regression=$(echo "$normalized_delta < -$tolerance" | bc -l 2>/dev/null || echo "0")
      if [ "$is_regression" = "1" ]; then
        regressed=$(echo "$regressed" | jq --arg name "$name" '. + [$name]')
      fi

      # Check improvement (normalized_delta > significance)
      local is_improvement
      is_improvement=$(echo "$normalized_delta > $significance" | bc -l 2>/dev/null || echo "0")
      if [ "$is_improvement" = "1" ]; then
        improved=$(echo "$improved" | jq --arg name "$name" '. + [$name]')
      fi
    done
  done

  # Apply set logic
  local verdict reason logic
  local regressed_count improved_count
  regressed_count=$(echo "$regressed" | jq 'length')
  improved_count=$(echo "$improved" | jq 'length')

  if [ "$regressed_count" -gt 0 ]; then
    verdict="regress"
    reason="regression in: $(echo "$regressed" | jq -r 'join(", ")')"
    logic="metric_regression_detected"
  elif [ "$improved_count" -gt 0 ]; then
    verdict="keep"
    reason="improved: $(echo "$improved" | jq -r 'join(", ")'), no regressions"
    logic="no_regressions_and_at_least_one_improvement"
  else
    verdict="neutral"
    reason="no meaningful changes detected"
    logic="no_improvements_above_significance"
  fi

  jq -n \
    --arg verdict "$verdict" \
    --arg reason "$reason" \
    --argjson metrics "$details" \
    --argjson improved "$improved" \
    --argjson regressed "$regressed" \
    --arg verdict_logic "$logic" \
    '{verdict: $verdict, reason: $reason, metrics: $metrics, improved: $improved, regressed: $regressed, verdict_logic: $verdict_logic}'
}
```

Update `# --- Main ---` to use scoring:

  echo "$gates_result"
  exit 0
}

metrics_result=$(run_benchmarks)

if [ "$INIT_MODE" = "true" ]; then
  jq -n \
    --argjson gates "$gates_result" \
    --argjson metrics "$metrics_result" \
    '{mode: "init", gates: $gates, metrics: $metrics}'
else
  scoring_result=$(score_metrics "$metrics_result" "$BASELINE")
  # Merge gates into scoring result
  echo "$scoring_result" | jq --argjson gates "$gates_result" '. + {gates: $gates}'

You are an experimenter agent for autoimprove. You have been spawned into an isolated git worktree to make a specific improvement to this codebase.

## Your Assignment

You will receive:
- **Theme**: what kind of improvement to attempt (e.g., "failing_tests", "todo_comments", "coverage_gaps", "lint_warnings")
- **Scope**: which files/directories to focus on (glob patterns)

1. **Explore** the codebase within your scope. Read relevant files. Understand the current state.
2. **Identify** a specific improvement that fits your theme.
3. **Implement** the change. Keep it focused and minimal.
4. **Verify** your changes work: run the test suite if available, check for syntax errors.

- Stay within your scope constraints (max files, max lines).
- Do not modify files in `forbidden_paths` (you'll be told which paths are forbidden).
- Test modifications must be **additive only** — you may add new tests but never delete or weaken existing assertions.
- Make exactly one commit with a clear message. Format: `<theme>: <what you did and why>`
- If you cannot find a meaningful improvement within your constraints, commit nothing and explain why.
- Do NOT try to discover or reverse-engineer how your changes are scored.

## Theme Guide

- **failing_tests**: Find and fix failing tests or the bugs they expose.
- **todo_comments**: Implement TODO/FIXME items found in the source code.
- **coverage_gaps**: Add tests for untested code paths.
- **lint_warnings**: Fix code quality issues, style problems, dead code.

### 2. Run

## What It Does

autoimprove picks from configured themes (failing tests, TODOs, coverage gaps, lint warnings) and spawns an experimenter agent into a git worktree. The experimenter is **blind to your benchmarks** — it makes changes it genuinely believes are improvements. A deterministic scoring script evaluates whether the change actually improved things.

Changes are kept only if:
- All gates pass (tests, type checker)
- No metric regresses beyond tolerance
- At least one metric improves beyond significance

This means autoimprove is conservative by default. It will never merge a change that makes your test suite fail or your benchmarks regress.

# Configuration Reference

autoimprove is configured via `autoimprove.yaml` in your project root.

## Full Schema

```yaml
project:
  name: my-project        # project name (for reports)
  path: .                  # project root (usually .)

budget:
  max_experiments_per_session: 20  # max experiments per /autoimprove run

gates:                     # hard gates — must all pass or experiment is discarded
  - name: tests
    command: npm test      # any command; exit 0 = pass
  - name: typecheck
    command: npx tsc --noEmit

benchmarks:                # metrics to track
  - name: my-benchmark
    type: script
    command: bash benchmark/metrics.sh
    metrics:
      - name: metric_name
        extract: "json:.field"     # json:<jq path> or shell command
        direction: higher_is_better  # or lower_is_better
        tolerance: 0.02            # max acceptable regression (2%)
        significance: 0.01         # min meaningful improvement (1%)

themes:                    # what the experimenter can try
  auto:
    strategy: weighted_random
    priorities:            # relative selection weight per theme
      failing_tests: 5
      todo_comments: 3
      coverage_gaps: 2
      lint_warnings: 2
    cooldown_per_theme: 3  # skip theme for N experiments after attempt

constraints:
  trust_ratchet:           # scope expands with demonstrated competence
    tier_0: { max_files: 3, max_lines: 150, mode: auto_merge }
    tier_1: { max_files: 6, max_lines: 300, mode: auto_merge, after_keeps: 5 }
    tier_2: { max_files: 10, max_lines: 500, mode: auto_merge, after_keeps: 15 }
    regression_penalty: -1 # drop one tier on any regression
  forbidden_paths:         # experimenter cannot touch these
    - autoimprove.yaml
    - benchmark/**
  test_modification: additive_only  # can add tests, not delete/weaken

safety:
  epoch_drift_threshold: 0.05      # halt if cumulative drift > 5%
  coverage_gate:                    # optional per-file coverage check
    command: "npx c8 report --reporter json"
    threshold: 0.80
  regression_tolerance: 0.02       # global default, overridable per metric
  significance_threshold: 0.01     # global default, overridable per metric
  stagnation_window: 5             # early stop after N non-improvements per theme
```

## Metric Extraction

The `extract` field tells evaluate.sh how to get a number from benchmark output:

- **`json:<jq_path>`** — parse JSON output with jq. Example: `json:.test_count`
- **Shell command** — pipe benchmark output through the command. Example: `grep -oP '\\d+(?= passed)'`

## Themes

Built-in theme types:
- `failing_tests` — find and fix failing tests or the bugs they expose
- `todo_comments` — implement TODO/FIXME items
- `coverage_gaps` — add tests for untested code paths
- `lint_warnings` — fix code quality issues
- `stale_code` — modernize outdated patterns

## Trust Ratchet

The system starts conservative (3 files, 150 lines) and earns scope:
- 5 consecutive keeps with zero regressions → tier 1 (6 files, 300 lines)
- 15 consecutive keeps → tier 2 (10 files, 500 lines)
- Any regression drops the tier by 1

# How autoimprove Works

## The Mental Model

autoimprove adapts [karpathy/autoresearch](https://github.com/karpathy/autoresearch) to software projects.

| autoresearch | autoimprove |
|---|---|
| `train.py` (agent edits this) | Your source code |
| `prepare.py` (immutable evaluation) | `evaluate.sh` (deterministic scoring) |
| `program.md` (human strategy) | `autoimprove.yaml` |
| `val_bpb` (fitness number) | Per-metric set logic |
| `git reset --hard` | `git worktree remove` |

## Architecture

**Two agents, one script:**

1. **Orchestrator** (Claude, main session) — reads config, picks themes, spawns experimenters, calls evaluate.sh, decides keep/discard, manages state
2. **Experimenter** (Claude, isolated worktree) — makes code changes, blind to scoring
3. **evaluate.sh** (deterministic bash+jq) — runs gates, benchmarks, extracts metrics, applies set logic

The experimenter never sees benchmark definitions, metric names, or scores. This prevents Goodhart's Law — the agent can't game a metric it can't observe.

## Scoring: Set Logic

No weighted composite. Each metric is independent:

- **No metric regresses** beyond its tolerance → safe
- **At least one metric improves** beyond its significance → valuable
- Both true → **KEEP**
- Regression detected → **DISCARD**
- No improvement → **NEUTRAL** (discard, increment stagnation)

## Safety Mechanisms

- **Epoch baseline**: frozen at session start, detects compound drift
- **Coverage gate**: changed files must maintain test coverage
- **Trust ratchet**: scope starts small, expands with success
- **Stagnation detection**: stops wasting budget on exhausted themes
- **Fast-forward merge only**: rebase failure = discard
- **Forbidden paths**: experimenter can't touch evaluation infrastructure
```
