# Approach C -- Enthusiast Review, Round 5 (Meta / Ergonomics / Ecosystem)

Rounds 1-4 covered architecture, implementation, completeness, and runtime robustness (21+ modifications accepted). This round examines the system from angles that code-level review cannot reach: the meta-question of whether this review process works for code, the actual user experience at scale, ecosystem interactions, and adversarial scoring.

---

## High

1. **HIGH: evaluate.sh spawns O(M*N) jq subprocesses per experiment, creating a quadratic bottleneck at scale.** Each metric comparison in `score_results` builds JSON incrementally: `SCORE_METRICS_JSON=$(echo "$SCORE_METRICS_JSON" | jq ...)`, `SCORE_REGRESSED=$(echo "$SCORE_REGRESSED" | jq ...)`, `SCORE_IMPROVED=$(echo "$SCORE_IMPROVED" | jq ...)`. That is 3 jq invocations per metric, plus 1 for the per-metric JSON object. With B benchmarks and M metrics each, a single scoring pass spawns ~4*B*M jq processes. At 5 benchmarks with 5 metrics each, that is 100 jq subprocesses per experiment. At 100 experiments per session, that is 10,000 jq forks for scoring alone. On macOS, `fork()+exec()` for jq takes ~5-10ms each, so scoring alone burns 50-100 seconds of wall time. The gate runner has the same pattern: `GATE_RESULTS=$(echo "$GATE_RESULTS" | jq ". + [$gate_json]")` -- one jq fork per gate. The fix is to accumulate in shell variables and emit a single jq call at the end of each loop, but the current implementation will measurably degrade session wall time as metric count grows.

2. **HIGH: The debate review process is structurally mismatched to evaluating code changes.** This is the philosophical question. The Enthusiast/Adversary/Judge pattern works well for design documents (like this very spec) because design decisions have debatable trade-offs with no ground truth. Code changes are different: they either pass tests or they do not, they either improve metrics or they do not. The deterministic evaluation via evaluate.sh already captures the objective signal. What remains for debate is subjective code quality -- maintainability, readability, architectural fit. But LLM-as-judge for subjective qualities is exactly what the design excludes from v1 ("No LLM Judge in v1"). The debate feature was bolted on as a superpower but its value proposition contradicts the core design principle. If debate output is truly "advisory annotation only" and never influences keep/discard, then it is a token-expensive comment generator whose output will be systematically ignored because the morning report already shows the objective signal. The design needs to honestly confront this: either debate influences decisions (breaking the no-LLM-judge principle) or it does not (making it expensive decoration).

3. **HIGH: experiments.tsv becomes unusable for human inspection past ~50 experiments.** The TSV format is designed for append-only simplicity, but a typical overnight session runs 18 experiments (per the morning report example). After 6 sessions, the file has 108+ rows. `cat experiments.tsv` produces a wall of text with no alignment (TSV columns are ragged). The morning report skill reads and summarizes it, but the design also points users directly to it: "Full log: ./experiments.tsv". A human opening this file in a terminal sees an unreadable mess. There is no `--last N` filter on the report command, no session markers in the TSV to delineate runs, and no rotation/archival strategy. The file grows monotonically and becomes less useful over time -- the opposite of what a log should do.

## Medium

4. **MEDIUM: The plugin has no mechanism to detect or handle its own CLAUDE.md being stale.** The orchestrator, experimenter, and all skills are Markdown prompts. Their behavior depends on CLAUDE.md project instructions being consistent with the actual plugin state. But if a user updates autoimprove (new version of orchestrator.md, changed evaluate.sh contract), CLAUDE.md may reference features, invariants, or file paths that no longer exist. There is no version assertion (e.g., "this orchestrator requires evaluate.sh schema >= 2") and no startup validation that the plugin files match the CLAUDE.md assumptions. This is a normal software versioning problem, but in an LLM plugin system where prompts are the code, stale instructions silently degrade behavior rather than producing compile errors.

5. **MEDIUM: The `set -uo pipefail` in evaluate.sh will hard-crash on any unset variable, but the error output goes to stderr while the orchestrator reads stdout.** The orchestrator expects a JSON verdict on stdout. If evaluate.sh crashes from an unset variable (e.g., a metric name typo in autoimprove.yaml produces a missing jq path), `set -u` kills the script, the error goes to stderr, and stdout is empty. The orchestrator gets an empty string instead of a JSON verdict. The design specifies "exit code is always 0 (verdict is in JSON)" but `set -u` can produce non-zero exits that bypass the structured JSON error handling. A `trap` that catches ERR and emits `{"verdict":"error","reason":"..."}` to stdout would close this gap.

6. **MEDIUM: Weighted random theme selection has no seed control, making sessions non-reproducible.** The orchestrator picks themes via weighted random. Two sessions with identical state will pick different theme sequences. This makes debugging impossible: "experiment 7 regressed" cannot be reproduced because the same experiment will not be selected in the same order. The implementation plan stores the theme in experiments.tsv but not the random seed. Adding a `--seed` flag (or logging the seed used) would allow exact session replay for debugging.

7. **MEDIUM: The additive-only test constraint is unenforceable at the evaluate.sh level.** The design says the experimenter "can add tests but cannot delete or weaken assertions." This is a prompt instruction to the LLM, not a programmatic check. evaluate.sh runs tests but does not diff test files to verify no assertions were removed. A malicious or confused experimenter could delete a failing test (making the gate pass) and add a trivial test (maintaining test count). The forbidden_paths mechanism protects specific paths, but test files are explicitly NOT forbidden -- the experimenter must edit them to add tests. Enforcing "additive only" requires a pre-merge check that no test file has fewer assertions than before, which is language-specific and unimplemented.

## Low

8. **LOW: Running autoimprove on the autoimprove repository itself creates a self-reference paradox with no defined behavior.** If a user runs `/autoimprove run` in the autoimprove repo, the experimenter could modify orchestrator.md, evaluate.sh, or the scoring logic itself. The forbidden_paths list protects autoimprove.yaml but not the plugin's own source files. An experimenter that "improves" evaluate.sh changes the evaluation criteria for the current and all future experiments. The design does not address self-hosting. For dogfooding, the plugin's own source should probably be in forbidden_paths, but this is not documented.

9. **LOW: Git worktrees interact poorly with other Claude Code plugins that use absolute paths.** If a user has plugins that reference files by absolute path (e.g., context-mode's FTS5 index, lcm's knowledge base), spawning an experimenter in a worktree at `/tmp/autoimprove-exp-001/` means those plugins either (a) index worktree-local files that will be deleted, polluting their databases, or (b) operate on the main repo path, not the worktree, producing inconsistent behavior. The design assumes the experimenter uses only basic Claude Code tools (Read, Edit, Bash), but the plugin system has no isolation boundary that prevents worktree-spawned agents from triggering other plugins.

10. **LOW: The trust ratchet's "consecutive keeps" counter is fragile to session boundaries.** The counter persists in state.json across sessions. But a new session resets the epoch baseline. If the previous session ended at tier 1 with 12 consecutive keeps, a new session starts with a fresh epoch baseline that may score differently. The first experiment might "regress" against the new baseline even though the code is identical -- simply because the benchmark is non-deterministic. One spurious regression drops the tier, punishing the user for starting a new session. The counter should arguably only count within a session, or the tier should be based on a rolling window rather than consecutive count.

---

I am scraping the barrel on items 8-10. The genuinely impactful findings are 1 (jq quadratic fork cost), 2 (philosophical mismatch of debate-for-code), and 3 (TSV ergonomics at scale). Items 4-7 are real but incremental. Beyond this list, I do not see findings that justify a round 6.
