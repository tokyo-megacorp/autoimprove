# Approach C -- Enthusiast Review, Round 3 (Deep Edge Cases)

Rounds 1-2 caught the architectural issues. This round goes deeper: interaction edge cases, design gaps that surface during real use, and subtle flaws in the challenge system.

---

## High

1. **HIGH: Debate findings accumulate across rounds with no deduplication or convergence mechanism.** The `rounds[]` array stores each round independently: `{ enthusiast_argument, adversary_argument, judge_verdict }`. In a 2-round debate, the Enthusiast in round 2 sees the round 1 verdict and may re-raise the same finding with different wording, or the Adversary may concede a point in round 1 but the Enthusiast repeats it in round 2 anyway. The Judge must rule on findings that may be semantically duplicated across rounds. There is no deduplication step, no "findings registry" that tracks which issues have been resolved vs. carried forward. The morning report consumer gets `rounds[0].judge_verdict` and `rounds[1].judge_verdict` with no guidance on which is authoritative. For the annotation in context.json, is the final verdict `rounds[N-1].judge_verdict`? What if the round-2 Judge contradicts the round-1 Judge? The `final_verdict` field in `debate_summary` is undefined in terms of derivation -- is it the last Judge verdict, a synthesis, or something else?

2. **HIGH: Challenge scoring has a partial-credit gap that distorts accuracy metrics.** The answer key schema is `{ file, line_range, defect_type, fix_pattern, severity }`. Matching is exact on `file + defect_type`, fuzzy on `line_range` (+/-5 lines). But what about partial matches? If a challenge has 5 planted bugs and the agent finds 3 correctly plus reports 2 false positives, the score is... undefined. Is it precision (3/5 found = 60%), recall (3/5 = 60%), F1, or just a raw count? False positive rate matters enormously for a bug-finder -- a tool that reports 20 "bugs" per file is useless even if 3 are real. The scoring spec defines field-level matching but not the aggregate scoring formula. Without this, challenge results are not comparable across runs or across agents.

3. **HIGH: `--rounds N` with N>2 has unbounded token cost with no marginal value signal.** The coarse budget enforcement skips remaining rounds if the sub-budget is exhausted. But the orchestrator has no way to detect convergence -- whether round 3 added anything round 2 did not. If the user sets `--rounds 5`, the system will burn through 5 rounds even if the Enthusiast, Adversary, and Judge are repeating themselves by round 3. There is no early-termination heuristic (e.g., "if round N judge verdict matches round N-1 verdict verbatim, stop"). The budget cap prevents runaway cost but not runaway waste within the cap.

4. **HIGH: Standalone `/autoimprove review` on a PR has no access to test results or benchmark data.** When debate runs inside the loop (post-verdict), the agents receive the diff plus the verdict/scores as context. When debate runs standalone via `/autoimprove review <PR>`, it operates on the diff alone. The user might expect the review to comment on whether the change would pass gates or improve benchmarks, but the standalone debate agents have no evaluate.sh integration, no baseline data, no metric context. The standalone and loop-integrated debate modes produce qualitatively different outputs with no indication to the user that they are operating under different information regimes.

---

## Medium

5. **MEDIUM: Challenge `fix_pattern` field has no specified semantics or matching algorithm.** The answer key includes `fix_pattern` but the scoring spec only defines matching on `file`, `line_range`, and `defect_type`. What is `fix_pattern` used for? Is it a regex matched against the agent's proposed fix? A human-readable description? A structural pattern like "add null check before dereference"? If it is decorative (documentation only), it should not be in the answer key schema. If it participates in scoring, the matching algorithm is unspecified. This field is either dead weight or an unimplemented feature masquerading as a schema field.

6. **MEDIUM: Challenge language coverage creates a combinatorial maintenance burden with unclear value.** Four languages (Python, TypeScript, Go, Rust) times N challenge packs means 4N sets of planted bugs, answer keys, and test harnesses. A single challenge ("off-by-one in loop") must be idiomatically planted in each language. But the Enthusiast/Adversary agents operate on source code via Claude -- they are language-agnostic by nature. If the point is to test whether the agents can find bugs in unfamiliar languages, one challenge per language suffices. If the point is to test bug-pattern coverage, multiple challenges in one language are more valuable than the same challenge in four languages. The 4-language requirement multiplies maintenance cost by 4x for unclear incremental signal.

7. **MEDIUM: The debate annotation in the morning report compresses multi-round output into a single line.** The specified format is `#ID verdict "key concern"`. For a 2-round debate where the Enthusiast raised 4 findings, the Adversary conceded 2 and rebutted 2, and the Judge ruled on each -- all of this is compressed to a single "key concern" string. The human reading the morning report has no way to gauge debate depth or disagreement level. They must open context.json to see whether the annotation represents unanimous agreement or heated disagreement. A 1-bit "verdict" plus 1-line "concern" loses the information that makes debate valuable over a single-pass review.

8. **MEDIUM: Trust ratchet interaction with debate is unspecified and creates a UX expectation gap.** At Tier 3 (propose-only, human reviews), the system already requires human approval for changes. The debate feature also generates human-facing review annotations. But there is no specification of how debate relates to the proposal review workflow. Does a Tier 3 proposal include its debate annotation? Does the debate verdict influence the proposal's "risk assessment"? If a proposal has a debate verdict of "significant concerns" but the metrics all pass, does the proposal queue surface this? The two human-facing review mechanisms (proposals + debate) operate in parallel with no integration point.

9. **MEDIUM: `/autoimprove challenge` has no baseline or regression tracking across runs.** The challenge command runs the agent against puzzles and reports scores. But there is no mechanism to store challenge results over time, detect score regressions (agent got worse at finding bugs after a model update), or compare scores across different agent configurations. The challenge system is a one-shot evaluation with no longitudinal tracking -- the opposite philosophy of the main autoimprove loop, which is all about tracking progress over time.

---

## Low

10. **LOW: Debate agents have no mechanism to request additional context.** The round 2 review identified the context-vs-budget tension (debate agents need context but context costs tokens). The resolution was that trust tiers bound diff size, making it manageable. But in standalone mode (`/autoimprove review <file>`), there are no trust tier bounds. A large file or a PR with 50 changed files could produce a diff that overwhelms the sub-budget before the Adversary even speaks. There is no mechanism for debate agents to say "I need to see the test file for this module" or "I need the function signature of the caller." They operate on whatever context they are given, which may be insufficient for a useful review.

11. **LOW: The Judge agent has a structural bias toward the last speaker.** In multi-round debate, the sequence is Enthusiast -> Adversary -> Judge, repeated N times. Within each round, the Adversary speaks last before the Judge rules, creating a recency bias. Across rounds, the final round's arguments are freshest in context. This is a known limitation of sequential debate formats, not unique to this design, but the spec does not acknowledge it or specify any mitigation (e.g., requiring the Judge to explicitly address each finding from both sides).

12. **LOW: Challenge difficulty calibration has no spec.** If all planted bugs are trivially obvious (missing null check on line 1), the challenge produces artificially high scores that do not predict real-world bug-finding ability. If all bugs are extremely subtle (race condition across 3 files), the challenge produces artificially low scores. The spec defines the answer key schema but not the difficulty distribution. Without calibration guidance, challenge authors will create packs of inconsistent difficulty, making cross-pack score comparisons meaningless.

13. **LOW: No specification for what happens when `/autoimprove review` is run with `--rounds N` and the input is a single-line diff.** A trivial diff (rename a variable, fix a typo) does not warrant multi-round adversarial debate. The system will dutifully run N rounds of Enthusiast/Adversary/Judge on a change that any single reviewer could assess in one pass. There is no complexity-based routing that says "this diff is trivial, force single-pass regardless of --rounds." The user can always pass `--single-pass`, but the default (2 rounds) will waste tokens on trivial reviews with no added value.
