# Approach C (Revised) -- Enthusiast Review, Round 2

**Focus:** Second-order implementation gaps, edge cases, UX pitfalls, challenge design flaws, scoring ambiguities. The obvious architectural issues (review_gate, insertion point, blindness, budget) are resolved. These are the things that will bite during implementation.

---

## Critical

1. **CRITICAL: Advisory annotation has no schema, no writer, and no reader.** The revised design says debate runs post-verdict and is "logged to experiment context." But `context.json` has a frozen schema (id, model_version, baseline_sha, experimenter_prompt, theme, scope, constraints, recent_experiments_provided, timestamp). There is no `debate_annotation` field. No specification of who writes it (orchestrator? a new post-processing step?). No specification of who reads it (the morning report? the human? future orchestrator sessions?). And critically: if nobody reads it programmatically, the debate output is write-only data -- it costs tokens to produce and delivers zero value unless the human manually inspects every `context.json`. The entire post-verdict annotation feature has no consumer.

2. **CRITICAL: `max_tokens_per_debate` has no enforcement mechanism.** The resolution says "add max_tokens_per_debate sub-budget counted against session budget." But Claude Code plugin agents have no token-counting API. The orchestrator cannot inspect how many tokens a spawned agent consumed mid-execution. It can set `max_tokens` on the agent spawn (if the plugin system supports it -- unverified), but it cannot enforce a sub-budget that is "counted against session budget" because there is no way to read back actual consumption from a completed agent and subtract it from a running total. The budget system in `autoimprove.yaml` tracks `max_cost_per_session` and `max_tokens_per_experiment` but the implementation plan (run/SKILL.md) shows no token-counting infrastructure at all -- budget checks are experiment-count based, not token-based.

3. **CRITICAL: Post-verdict debate creates a timing/ordering problem with experiment logging.** The loop flow is: SCORE -> KEEP/DISCARD -> merge/delete worktree -> log to experiments.tsv -> update cooldowns. If debate runs post-verdict, it needs the worktree content (the diff, the changed files) to have anything to review. But if the verdict is DISCARD, the worktree is deleted before debate can run. Either: (a) debate must run before worktree cleanup, which means it runs between verdict and worktree deletion -- inserting logic into a sequence the design treats as atomic, or (b) the diff must be captured and stored before worktree deletion so debate can run on it later -- adding a new "diff snapshot" artifact that is not in the design. Neither option is specified.

## High

4. **HIGH: Challenge answer key comparison is an unsolved problem in disguise.** The revised design specifies "curated puzzles with planted bugs and answer keys." But bug-finding is not a classification task with exact answers. A planted bug might be "off-by-one in loop bound on line 47." The agent might report "potential infinite loop on line 47" or "boundary condition error in the for loop starting at line 45" or "array index out of bounds near line 48." All three are correct identifications of the same bug, expressed differently. Comparing agent output to answer keys requires either: (a) exact string/line matching (brittle, high false-negative rate), (b) LLM-as-judge to assess semantic equivalence (contradicts no-LLM-judge in scoring), or (c) a structural format where agents must output `{file, line_range, bug_category}` and comparison is field-level (requires a rigid output schema that constrains agent behavior). Option (c) is probably correct but is entirely unspecified and fundamentally shapes the agent prompt design.

5. **HIGH: GitHub case studies cannot be reproduced deterministically.** The design mentions "GitHub case studies" as part of the challenge corpus. Real GitHub diffs and issues change over time (repos get deleted, force-pushed, reorganized). If challenges reference external GitHub content by URL, the challenge suite is non-reproducible. If the content is vendored locally, it is a maintenance burden and licensing concern. And unlike planted puzzles with known answers, real GitHub bugs have ambiguous ground truth -- was the fix correct? Was it the only valid fix? Was there a deeper root cause? The "answer key" for a real-world bug is inherently debatable, which makes deterministic scoring impossible.

6. **HIGH: `--single-pass` and `--rounds N` produce structurally different outputs with no unified interface.** Single-pass produces one Enthusiast report + one Adversary report + one Judge verdict. Multi-round produces N of each. The consumer of debate output (the annotation writer, the morning report, the human reader) must handle both shapes. Is the final verdict in single-pass the same schema as the round-N verdict in multi-round? When multi-round is used, is only the final round's verdict recorded, or all rounds? If all rounds, the annotation data model is an array of variable length. None of this is specified and getting it wrong means the annotation is unparseable.

7. **HIGH: Debate agents operating on a diff have an asymmetric information problem.** The Enthusiast and Adversary receive the diff, but the quality of a change depends heavily on what the code looked like before, what the surrounding architecture is, what the project conventions are, and what the tests cover. A 10-line diff in isolation is often ambiguous. The experimenter had full project context (it was working in the worktree). The debate agents need comparable context to produce useful reviews, but providing "full project context" to three agents across N rounds is a token cost multiplier that `max_tokens_per_debate` is supposed to cap. There is a fundamental tension between review quality (needs context) and budget (context is expensive), and no guidance on how to resolve it.

8. **HIGH: No specification of what "advisory annotation only" means in the morning report.** The morning report template shows kept/discarded experiments with scores and commit messages. Where does the debate annotation appear? Is it a new column? A sub-section? A separate file? If the human is supposed to read debate output during morning review, the report template needs to surface it. If it is buried in `context.json`, it will never be read. The entire value proposition of post-verdict debate is that a human reads it, and the primary human touchpoint (the morning report) has no integration point.

## Medium

9. **MEDIUM: Challenge suite "multi-language" support requires per-language test harnesses.** A planted bug in Python needs pytest to verify the fix. One in TypeScript needs jest or vitest. One in Rust needs cargo test. The challenge runner must detect the language, install dependencies, and run the correct test command. This is a project-setup problem equivalent to `/autoimprove init` but for each challenge puzzle. The implementation scope is closer to "build a polyglot CI system" than "curate some puzzles."

10. **MEDIUM: Debate agent tool access is unspecified and has security implications.** The experimenter agent has explicit tool access: Read, Write, Edit, Glob, Grep, Bash. Debate agents need Read and Grep to examine code but should NOT have Write, Edit, or Bash -- they must not modify the worktree or execute arbitrary commands on the host. But agent tool access in Claude Code plugins is defined in the agent frontmatter. If debate agents are given Bash for legitimate reasons (e.g., running `git diff`), they could accidentally or intentionally modify state. The tool access matrix for three new agents is a security-relevant design decision that is completely absent.

11. **MEDIUM: "Debate runs post-verdict" has no error handling or timeout specification.** If the Enthusiast agent hangs, crashes, or produces malformed output mid-debate, what happens? The verdict is already finalized, so the experiment outcome is unaffected -- but the session is now stuck waiting for a debate that will never finish. Is there a per-agent timeout? Does the orchestrator continue the loop if debate fails? Is a failed debate logged differently than a successful one? The "advisory only" framing makes it easy to dismiss error handling, but a hung agent blocks the entire session.

12. **MEDIUM: Planted bug difficulty calibration has no methodology.** "Curated puzzles" implies someone decides what bugs to plant and how hard they should be. But difficulty is subjective and model-dependent -- a bug that stumps Claude Sonnet might be trivial for Claude Opus. Without a difficulty rating system, a calibration methodology (run N models, record pass rates, assign difficulty tiers), and periodic recalibration as models improve, the challenge suite cannot meaningfully track agent capability over time. It becomes a binary pass/fail with no granularity.

13. **MEDIUM: The `--rounds N` flag implies each round's agents see previous rounds' output, creating compounding context growth.** Round 1: Enthusiast produces ~2K tokens, Adversary ~2K, Judge ~1K. Round 2: Enthusiast sees round 1 output (~5K) + original diff, produces ~3K. Adversary sees all of that (~8K), produces ~3K. By round 3, each agent's input context includes the full debate history (~15-20K tokens). With `max_tokens_per_debate` set to, say, 30K, you can afford maybe 2 rounds before hitting the cap. The default `--rounds N` must be informed by this math, but the math depends on the unspecified context-sharing strategy (#25 from R1, still unresolved).

14. **MEDIUM: No distinction between "debate found a real issue the deterministic gates missed" and "debate produced noise."** When debate output is logged as an annotation, there is no mechanism to later validate whether the Enthusiast's concerns were legitimate. If experiment #007 is kept (passed all gates, improved score) and the Enthusiast flagged a potential memory leak, was the Enthusiast right? The only way to know is if the memory leak manifests later -- but there is no feedback loop connecting future failures to past debate annotations. Without this, debate annotations accumulate but never self-correct.

15. **MEDIUM: `/autoimprove review` on arbitrary code has no connection to the project's `autoimprove.yaml` configuration.** When used standalone on a PR or diff, the debate agents have no knowledge of the project's benchmarks, gates, themes, or scoring criteria. They review code in a vacuum. This means standalone review quality depends entirely on the agents' general code review ability, not on any project-specific calibration. The "parallel tracks" framing implies these tools complement the loop, but standalone review is a generic code review tool that happens to share agent definitions -- it gains nothing from the autoimprove infrastructure.

16. **MEDIUM: Session budget accounting for post-verdict debate is ambiguous when the session ends mid-debate.** The loop checks budget at the top: "Check budget (tokens, time, cost) -> exhausted? STOP." If the session budget is nearly exhausted and a debate is running post-verdict, does the debate get killed mid-stream? Does it finish (potentially exceeding budget)? Is there a reserve allocation to ensure debate can always complete? The budget check happens before the experiment, but the debate cost is incurred after -- creating a window where actual spend exceeds the configured cap.

---

## Low

17. **LOW: "Curated puzzles" and "GitHub case studies" are two fundamentally different challenge types that need different runners.** Puzzles have a known answer and a verification script. GitHub case studies have a real commit history and debatable ground truth. Lumping them under `--suite puzzles|github|all` implies a unified scoring framework, but the evaluation criteria are incompatible. Puzzle scoring is precision/recall against answer keys. GitHub scoring is... what? Agreement with the actual fix? But the actual fix might not be the best fix. These need separate scoring systems, not a shared `--suite` flag.

18. **LOW: No versioning strategy for the challenge corpus.** If puzzles are updated (bugs changed, new ones added, old ones retired), historical challenge scores become incomparable. Score of 8/10 on v1 of the corpus is not the same as 8/10 on v2. There is no corpus version identifier, no changelog, and no guidance on when/how to update puzzles while preserving comparability.

19. **LOW: The Adversary agent's effectiveness depends on the Enthusiast's output quality, creating a fragile chain.** If the Enthusiast misses a real issue, the Adversary has no way to surface it -- the Adversary only responds to the Enthusiast's claims. This means the debate's recall is bounded by the Enthusiast's recall. A single-agent reviewer with both roles might actually have better coverage than a debate where the Adversary is anchored to the Enthusiast's (potentially incomplete) findings.

20. **LOW: No specification of whether `/autoimprove challenge` results feed into `/autoimprove review` agent selection or configuration.** If challenges reveal that the Enthusiast agent is poor at finding concurrency bugs but excellent at finding type errors, should `/autoimprove review` adjust the Enthusiast's prompt or focus? The challenge suite benchmarks capability but there is no mechanism to act on the results -- they are purely observational.

21. **LOW: Post-verdict debate on DISCARD experiments is lower-value than on KEEP experiments.** A discarded experiment's changes never reach main -- the debate annotation describes issues with code that was already rejected. The primary value case for debate is catching subtle issues in KEPT experiments that passed deterministic gates. But the design does not distinguish: debate runs on all experiments equally, wasting budget on changes that were already rejected by harder gates.

22. **LOW: The revised approach says "debate output excluded from experimenter prompt template" but there is no specification of what IS in the experimenter prompt template.** The experimenter agent frontmatter (`experimenter.md`) defines the agent's personality and rules, but the actual per-experiment prompt (theme, scope, constraints, recent history) is constructed by the orchestrator skill at runtime. The exclusion of debate output is a negative constraint ("do not include X") on a template that is not positively specified. Without a positive spec of the prompt template, it is impossible to verify the exclusion.
